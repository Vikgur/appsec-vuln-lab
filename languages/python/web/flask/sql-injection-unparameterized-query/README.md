# Уязвимость

SQL Injection (CWE-89) — уязвимость, возникающая при формировании SQL-запросов через небезопасную конкатенацию строк с пользовательским вводом.

Атакующий может внедрить собственный SQL-код в запрос приложения и изменить логику выполнения запроса.

Классификация:

* OWASP Top-10: A03 – Injection
* CWE: CWE-89 – Improper Neutralization of Special Elements used in an SQL Command

# Атакуемая поверхность

Типичные точки возникновения уязвимости:

* HTTP query parameters
* параметры форм (login / search)
* фильтры и сортировки
* REST API параметры
* любые внешние данные, передаваемые в SQL-запрос

# Уязвимый код

В уязвимой версии приложения SQL-запрос формируется через прямую конкатенацию строки с пользовательским вводом.

Проблема:

* пользовательский ввод попадает в SQL без экранирования
* отсутствуют prepared statements
* отсутствует параметризация запроса

Это позволяет атакующему изменить структуру SQL-запроса.

# Эксплуатация

Файл `exploit_payload.txt` содержит пример полезной нагрузки.

Пример payload:

' OR 1=1 --

Результат:

* условие WHERE становится истинным
* база возвращает все записи

Типовые техники эксплуатации:

* authentication bypass
* UNION based extraction
* boolean-based injection

# Влияние

Потенциальные последствия:

* обход авторизации
* чтение данных из базы
* изменение или удаление данных
* раскрытие структуры базы

В критических случаях возможно дальнейшее развитие атаки до:

* privilege escalation
* remote command execution через функции СУБД

# Корневая причина

Основные инженерные причины:

* небезопасная конкатенация строк
* отсутствие параметризованных запросов
* отсутствие безопасных API для работы с БД
* недостаточный контроль пользовательского ввода

# Исправление

Правильный security-паттерн — использование параметризованных запросов.

Безопасные подходы:

* prepared statements
* параметризованные запросы
* использование ORM
* строгая валидация входных данных

Основной принцип:

Пользовательский ввод никогда не должен напрямую влиять на структуру SQL-запроса.

# Обнаружение

Уязвимость может быть обнаружена автоматически.

SAST:

* правило статического анализа обнаруживает небезопасное формирование SQL-запроса
* анализируется поток данных от пользовательского ввода к SQL

DAST:

* автоматическое сканирование HTTP endpoint
* подстановка SQL payload
* анализ поведения приложения

# Интеграция в DevSecOps

Автоматические проверки выполняются в security pipeline.

Типовые этапы:

1. pre-commit проверки
2. SAST анализ исходного кода
3. security stage в CI/CD
4. DAST сканирование запущенного приложения

При обнаружении уязвимости пайплайн может блокировать merge или релиз.

# Предотвращение регрессии

Чтобы предотвратить повторное появление уязвимости:

* добавляются правила статического анализа
* вводятся security-gates в CI/CD
* выполняется регулярное сканирование приложения

Таким образом любая новая попытка добавить небезопасный SQL будет автоматически обнаружена.
