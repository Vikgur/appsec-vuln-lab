# Чек-лист по детектированию SQL 

Чек-лист для разработчиков перед написанием кода, работающего с базой данных.

---

## 1. Никогда не формировать SQL через конкатенацию строк

Запрещено включать пользовательский ввод напрямую в SQL-запрос.

Плохо:
`"SELECT * FROM users WHERE name = '" + user_input + "'"`

---

## 2. Всегда использовать параметризованные запросы

Все динамические SQL-запросы должны использовать **prepared statements / parameter binding**.

Пример безопасного подхода:
`SELECT * FROM users WHERE name = ?`

---

## 3. Предпочитать ORM или безопасные database API

Если используется ORM:

* использовать стандартные методы ORM
* избегать `raw SQL` и `execute` без необходимости

Если используется SQL напрямую:

* применять только параметризованные запросы.

---

## 4. Валидировать пользовательский ввод

Любые данные из HTTP-запросов считаются недоверенными.

Минимальные проверки:

* ограничение **длины строк**
* проверка **формата** (email, UUID, дата)
* использование **белых списков** для ограниченных значений

Пример:
`status ∈ {active, inactive}`

---

## 5. Безопасная работа с динамическими частями SQL

Некоторые части SQL нельзя параметризовать:

* `ORDER BY`
* `LIMIT`
* `OFFSET`
* имена колонок
* имена таблиц

Для таких случаев:

* использовать **allowlist допустимых значений**.

---

## 6. Не раскрывать SQL-ошибки пользователю

Пользователь не должен видеть:

* текст SQL-запроса
* структуру базы
* названия таблиц
* стек-трейсы

Показывать только общее сообщение об ошибке.

---

## 7. Проверять SQL-код перед коммитом

Перед отправкой кода убедиться:

* нет конкатенации SQL-строк
* используются параметризованные запросы
* ввод валидируется
* отсутствует небезопасный raw SQL
