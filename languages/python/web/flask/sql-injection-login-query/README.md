# Уязвимость

**SQL Injection (CWE-89)** — уязвимость, возникающая при формировании SQL-запросов с использованием пользовательского ввода без безопасной параметризации.

Если пользовательский ввод напрямую влияет на структуру SQL-запроса, атакующий может внедрить собственный SQL-код и изменить логику выполнения запроса.

Классификация:

* **OWASP Top 10:** A03 – Injection
* **CWE:** CWE-89 – Improper Neutralization of Special Elements used in an SQL Command

SQL Injection остаётся одной из наиболее распространённых и критичных уязвимостей веб-приложений.

---

# Атакуемая поверхность

SQL Injection может возникать в любых местах, где пользовательский ввод используется при формировании SQL-запросов.

Типичные точки возникновения:

* HTTP query parameters
* параметры форм (login, search, filters)
* REST API параметры
* параметры сортировки и фильтрации
* JSON payload API
* любые внешние данные, передаваемые в SQL-запрос

Особенно часто уязвимость появляется в:

* login endpoints
* search endpoints
* user lookup API
* admin dashboards

---

# Границы доверия и точки входа

**Границы доверия** определяют, где происходит переход между зонами с разным уровнем доверия:  
* пользователь → backend (входные данные от клиента не считаются доверенными)
* backend → БД (SQL‑запросы формируются с учётом возможного вредоносного ввода)
* pod → metadata (доступ к метаданным кластера ограничивается по принципу наименьших привилегий)

**Точки входа** — конкретные интерфейсы, через которые поступает внешний ввод:  
* HTTP‑эндпоинты: `/login`, `/search`, `/api/v1/users`
* параметры запросов: `?username=...`, `?query=...`
* тела запросов: JSON‑payload в `POST /api/v1/data`
* заголовки HTTP (в случаях, когда они влияют на логику обработки)

Для данного кейса критичные точки входа:  
* `POST /login` (поля `username` и `password`);
* любые параметры, подставляемые в SQL‑запросы без параметризации

# Описание фичи кейса

Кейс моделирует типичную функцию **аутентификации пользователя** в backend-приложении на **Python Flask**.

Фича:

`POST /login`

Пользователь отправляет:

* username
* password

Сервер выполняет SQL-запрос к базе данных для проверки учетных данных.

Подобная логика используется в:

* SaaS приложениях
* корпоративных админ-панелях
* CRM / ERP системах
* внутренних сервисах

Ошибка возникает при формировании SQL-запроса через строковую интерполяцию.

# Уязвимая реализация

В уязвимой версии SQL-запрос формируется напрямую из пользовательского ввода.

Проблема:

* пользовательский ввод вставляется в SQL-запрос
* отсутствуют prepared statements
* отсутствует параметризация

Пример небезопасного подхода:

```
query = f"SELECT * FROM users WHERE username = '{username}' AND password = '{password}'"
cursor.execute(query)
```

Поскольку пользовательский ввод становится частью SQL-запроса, атакующий может изменить его структуру.

# Эксплуатация

Файл `exploit_payload.txt` содержит примеры полезной нагрузки.

Пример payload:

```
' OR 1=1 --
```

Результат:

* условие `WHERE` становится всегда истинным
* база данных возвращает первую найденную запись

Это приводит к **обходу аутентификации (login bypass)**.

Распространённые техники SQL Injection:

* authentication bypass
* UNION-based data extraction
* boolean-based injection
* error-based injection

---

# Влияние

Последствия SQL Injection могут быть критичными.

Возможные воздействия:

* обход аутентификации
* доступ к чужим аккаунтам
* чтение конфиденциальных данных
* изменение данных в базе
* удаление данных

Дополнительные риски:

* раскрытие структуры базы данных
* privilege escalation
* использование функций СУБД для дальнейшей атаки

В production-системах SQL Injection может привести к **полной компрометации базы данных**.

# Корневая причина

Основные инженерные причины:

* использование строковой конкатенации для SQL
* отсутствие prepared statements
* неправильное использование database API
* отсутствие secure coding практик

Системные причины:

* отсутствие SAST-проверок
* отсутствие security-review
* отсутствие DevSecOps-pipeline

Ключевая ошибка:

**пользовательский ввод влияет на структуру SQL-запроса.**

---

# Исправление

Основной способ защиты — **параметризованные запросы (prepared statements)**.

Безопасный пример:

```
query = "SELECT * FROM users WHERE username = ? AND password = ?"
cursor.execute(query, (username, password))
```

Преимущества:

* пользовательский ввод отделён от SQL-кода
* база данных интерпретирует ввод только как данные

Дополнительные меры защиты:

* использование ORM
* строгая валидация входных данных
* принцип least privilege для БД
* Web Application Firewall

# Обнаружение

SQL Injection может быть обнаружена автоматически.

### SAST

Статический анализ исходного кода:

* обнаружение небезопасной интерполяции SQL
* анализ потоков данных (user input → SQL query)

Инструменты:

* Semgrep
* CodeQL
* Bandit

### DAST

Динамическое тестирование приложения:

* отправка SQL payload
* анализ реакции приложения

Инструменты:

* OWASP ZAP
* Nuclei

### Fuzzing

Автоматическая генерация payload:

* тестирование API параметров
* поиск уязвимых точек ввода

Инструменты:

* RESTler
* ffuf

# Интеграция в DevSecOps

SQL Injection должен обнаруживаться автоматически в security pipeline.

Типичный DevSecOps процесс:

1. **pre-commit**

   * SAST анализ кода

2. **CI pipeline**

   * запуск SAST инструментов
   * security linting

3. **security stage**

   * DAST сканирование API

4. **pre-release**

   * fuzzing и security testing

При обнаружении SQL Injection:

* merge блокируется
* релиз останавливается

# Предотвращение регрессии

Чтобы исключить повторное появление уязвимости:

* внедряются правила статического анализа
* добавляются security-gates в CI/CD
* выполняется регулярное security-сканирование
* проводится secure code review

Это позволяет автоматически обнаруживать любые новые попытки небезопасного SQL.
