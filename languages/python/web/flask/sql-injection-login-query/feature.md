## Продуктовый контекст

Типичный SaaS-продукт или внутренний корпоративный портал использует Python + Flask для быстрого создания backend API.
Одной из первых реализуемых функций почти всегда является **аутентификация пользователя через форму входа**.

Фича:
POST `/login` — пользователь отправляет `username` и `password`, после чего сервер проверяет их в базе данных.

Такие реализации часто встречаются в:

* банковских клиентских порталах
* внутренних админ-панелях
* CRM / ERP системах
* SaaS-сервисах
* MVP-стартапах на Flask

## Как возникает уязвимость

Разработчик формирует SQL-запрос через **строковую конкатенацию**, напрямую вставляя пользовательский ввод в SQL.

Причины:

* быстрый MVP
* отсутствие ORM
* незнание параметризованных запросов
* копирование примеров из StackOverflow

В результате злоумышленник может внедрить собственный SQL-код в поле `username` или `password`.

## Реальный сценарий атаки

Атакующий отправляет специальный payload в поле `username`:

' OR 1=1 --

SQL-запрос превращается в:

SELECT * FROM users WHERE username='' OR 1=1 --' AND password='anything'

Условие `OR 1=1` всегда истинно.

Результат:

* обход аутентификации
* вход в систему без пароля
* потенциальный доступ к админ-панели

## Почему этот кейс важен

Это **одна из самых известных и до сих пор встречающихся уязвимостей**.

Она часто появляется в:

* login endpoints
* search endpoints
* ID lookup API
* админ-панелях

Именно **SQL Injection в login endpoint** регулярно используется в обучающих демонстрациях, CTF и реальных атаках.
