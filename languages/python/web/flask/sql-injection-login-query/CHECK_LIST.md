# Чек-лист безопасной реализации SQL в Python / Flask

Практический чек-лист для разработчиков перед написанием или ревью кода, работающего с базой данных.

Контекст кейса: **login endpoint Flask API**, взаимодействующий с SQL-базой данных.

---

# 1. Никогда не формировать SQL через конкатенацию строк

Пользовательский ввод **никогда** не должен напрямую вставляться в SQL-запрос.

Запрещённые паттерны:

* строковая конкатенация
* f-строки
* string formatting (`%`)

Плохо:

```
query = f"SELECT * FROM users WHERE username = '{username}'"
```

или

```
query = "SELECT * FROM users WHERE username = '" + username + "'"
```

Такие конструкции позволяют атакующему изменить структуру SQL-запроса.

---

# 2. Всегда использовать параметризованные запросы

Все динамические SQL-запросы должны использовать **prepared statements / parameter binding**.

Правильный подход:

```
query = "SELECT * FROM users WHERE username = ? AND password = ?"
cursor.execute(query, (username, password))
```

Преимущества:

* пользовательский ввод передается отдельно от SQL-кода
* СУБД интерпретирует ввод только как данные

Это **основная защита от SQL Injection**.

---

# 3. Предпочитать ORM вместо raw SQL

При разработке backend-сервисов рекомендуется использовать ORM.

Примеры для Python:

* SQLAlchemy
* Django ORM

Преимущества ORM:

* автоматическая параметризация запросов
* меньше ручной работы с SQL
* меньше риска SQL Injection

Рекомендации:

* избегать `raw SQL` без необходимости
* избегать `execute()` с динамическими строками

---

# 4. Валидировать пользовательский ввод

Все данные из HTTP-запросов считаются **недоверенными**.

Для login endpoint рекомендуется:

Минимальные проверки:

* ограничение длины `username`
* ограничение длины `password`
* допустимый набор символов
* проверка формата (email / username policy)

Пример:

```
username length ≤ 64
password length ≤ 128
```

Важно:

валидация **не заменяет параметризацию**, а используется вместе с ней.

---

# 5. Безопасная работа с динамическими частями SQL

Некоторые части SQL **невозможно параметризовать**:

* `ORDER BY`
* `LIMIT`
* `OFFSET`
* имена таблиц
* имена колонок

Для таких случаев необходимо использовать **allowlist допустимых значений**.

Пример:

```
allowed_sort = ["name", "created_at"]
if sort not in allowed_sort:
    reject_request()
```

Никогда не передавать такие значения напрямую из пользовательского ввода.

---

# 6. Использовать принцип least privilege для пользователя БД

Аккаунт базы данных, используемый приложением, должен иметь **минимально необходимые права**.

Для login endpoint достаточно:

* `SELECT` для таблицы пользователей

Запрещено:

* `DROP`
* `ALTER`
* `CREATE`
* `SUPERUSER` права

Это снижает влияние возможной эксплуатации уязвимости.

---

# 7. Не раскрывать SQL-ошибки пользователю

Сообщения об ошибках не должны раскрывать:

* SQL-запросы
* структуру базы данных
* названия таблиц
* stack traces

Плохо:

```
sqlite3.OperationalError: near "OR": syntax error
```

Правильный подход:

```
Authentication failed
```

Детали ошибки должны логироваться **только на сервере**.

---

# 8. Использовать безопасное логирование

При логировании ошибок:

* не записывать SQL-запросы с пользовательскими данными
* не логировать пароли
* не логировать полный payload запроса

Безопасный лог:

```
Login attempt failed for user
```

---

# 9. Использовать DevSecOps-сканирование

SQL Injection должен обнаруживаться автоматически.

Рекомендуемые классы инструментов:

**SAST**

* Semgrep
* CodeQL
* Bandit

Обнаруживают:

* небезопасное формирование SQL
* поток данных user input → SQL query

**DAST**

* OWASP ZAP
* Nuclei

Обнаруживают:

* уязвимые HTTP endpoints
* успешную эксплуатацию SQL payload

**Fuzzing**

* RESTler
* ffuf

Проверяют:

* параметры API
* неожиданные payload

---

# 10. Включить security-гейты в CI/CD

В pipeline должны быть security-проверки.

Минимальный процесс:

1. pre-commit SAST
2. security stage в CI
3. DAST тестирование API
4. fuzzing перед релизом

При обнаружении SQL Injection:

* merge должен блокироваться
* релиз должен останавливаться.

---

# 11. Проводить security code review

Перед merge необходимо проверить:

* отсутствует SQL-конкатенация
* используются parameterized queries
* нет raw SQL без необходимости
* входные данные валидируются

---

# 12. Проводить регулярное security-тестирование

Для production-систем:

* регулярные DAST-сканирования
* bug bounty программы
* penetration testing

Это помогает обнаружить уязвимости, пропущенные в разработке.

---

# Итоговый принцип

Основное правило защиты от SQL Injection:

**Пользовательский ввод никогда не должен влиять на структуру SQL-запроса.**

Безопасная архитектура достигается комбинацией:

* parameterized queries
* ORM
* input validation
* least privilege DB access
* DevSecOps security scanning
* secure code review.
